## 1. Why Metrics Matter for This Programme

OpenMRS Core is a large, stable healthcare system.
The modernisation programme (Java 8 patterns → Java 21) touches hundreds of files across a 158,000-line codebase.
Without baseline measurements:

- It is impossible to know whether refactoring is *improving* structural health or just moving code around.
- Silent regressions accumulate (new god files, additional broad catches) without anyone noticing.
- Stakeholders cannot evaluate whether Phase 1 work achieved its goals before Phase 2 begins.

**Guiding principle**: Every metric tracked here exists because it answers a specific quality question, not for its own sake.

| Quality Question | Answered By |
|---|---|
| Are we using Java 21? | Java 21 adoption counters |
| Is the codebase becoming less tangled? | God-file count, boolean-param count |
| Are we hiding failures? | Broad catch / silent catch counts |
| Are tests keeping pace with changes? | JaCoCo line and branch coverage |
| Are static analysers catching new problems? | SpotBugs finding count |
| Is the build staying healthy? | Test pass rate, build time |

## 2. How to Capture Metrics

### 2.1 Static analysis (fast, no build needed)

```bash
# Run from repository root
bash tools/capture-baseline-metrics.sh
```

Outputs a Markdown table to stdout. Use `--format json` for CI artefacts.

```bash
# Save a named snapshot for later comparison
bash tools/capture-baseline-metrics.sh \
    --snapshot docs/snapshots/metrics-$(date +%Y%m%d).md

# Compare current state against the saved snapshot
bash tools/capture-baseline-metrics.sh \
    --diff docs/snapshots/metrics-20250101.md
```

See `bash tools/capture-baseline-metrics.sh --help` for all options.

### 2.2 Coverage (requires a build)

JaCoCo reports are generated by:

```bash
./mvnw jacoco:prepare-agent verify jacoco:report -B -ntp -Dspotbugs.skip=true
```

Reports land in `<module>/target/site/jacoco/index.html` for each module.
The CI pipeline (`ci-modernisation.yml`) runs this automatically and uploads reports as build artefacts.

### 2.3 SpotBugs (requires compiled bytecode)

```bash
./mvnw compile spotbugs:check -B -ntp
```

The number of findings is printed to console and fails the build if any violations breach
the configured severity threshold. The suppression list lives in `spotbugs-exclude.xml`.

### 2.4 Checkstyle (source only)

```bash
./mvnw checkstyle:check -B -ntp
```

The rule set is in `checkstyle.xml`. Most rules are configured at `warning` severity;
a subset are promoted to `error` and will fail the build.

### 2.5 Performance (manual capture only)

Application startup time and endpoint latency require a running environment.
See [Section 3.5](#35-performance) for the capture methodology.

## 3. Metrics Reference

### 3.1 Code Quality — SpotBugs & Checkstyle

#### What we capture

| Metric | How captured | Authoritative? |
|---|---|---|
| SpotBugs total findings | `spotbugs:check` output | ✅ Yes |
| SpotBugs findings by category | SpotBugs HTML report | ✅ Yes |
| Checkstyle error-severity violations | `checkstyle:check` output | ✅ Yes |
| Checkstyle warning-severity violations | Checkstyle HTML report | ✅ Yes |

#### Why it matters

SpotBugs performs bytecode-level analysis and catches real bugs that grep cannot:
null-pointer dereferences, resource leaks, incorrect synchronisation, and security patterns.
Checkstyle enforces structural conventions that keep the codebase readable across a large contributor base.

#### How to interpret

- **SpotBugs findings** should trend toward zero. Any new finding on a PR is a gate failure.
- The suppression file (`spotbugs-exclude.xml`) contains known legacy issues temporarily excluded.
  As those files are refactored, their suppressions should be removed. A growing suppression file
  is itself a regression signal.
- **Checkstyle error violations** block merges. Warning violations are informational but should not grow.

#### Phase targets

| Metric | Baseline | Phase 1 Target | Phase 3 Target |
|---|---|---|---|
| SpotBugs findings (new, unsuppressed) | 0 on CI | 0 | 0 |
| Suppression entries in `spotbugs-exclude.xml` | Audit pending | ≤ baseline | Reduce by 20% |
| Checkstyle error violations | 0 on CI | 0 | 0 |

### 3.2 Test Coverage — JaCoCo

#### What we capture

| Metric | Scope | How captured |
|---|---|---|
| Line coverage % | Per module (`api`, `web`) | JaCoCo HTML/XML report |
| Branch coverage % | Per module | JaCoCo HTML/XML report |
| Missed lines | Per module | JaCoCo report |
| Missed branches | Per module | JaCoCo report |

#### Why it matters

Coverage is the primary safety net for refactoring.
Without coverage data we cannot tell whether a code restructuring that passes all tests does so because:
(a) the tests validated it is correct, or (b) the tests simply don't cover the changed code.

The `api` module has the greatest coverage risk because it contains most of the domain logic
and is the primary target of Phase 1.5 god-class decomposition.

#### How to interpret

- **Line coverage** measures which lines were executed by at least one test.
  It is a floor, not a ceiling — 80% line coverage can still leave critical branches untested.
- **Branch coverage** measures whether both the true and false branches of every conditional
  were exercised. This is more meaningful for quality refactoring.
- A drop in coverage after a PR means either (a) new code was added without tests, or
  (b) existing tests were deleted. Both require justification.

### 3.3 Java 21 Feature Adoption

#### What we capture

These counters are collected by `tools/capture-baseline-metrics.sh` using text-pattern analysis.
They are approximate (textual matching only, not semantic) but directionally reliable.

| Feature | Pattern matched | Why tracked |
|---|---|---|
| `var` local type inference | `\bvar [a-zA-Z_$]` | Readability uplift; Phase 1 low-risk change |
| Records | `record\s+[A-Z]` at declaration | New Java 16+ feature; zero adoption today |
| Sealed classes | `\bsealed\b.*class\|interface` | New Java 17+ feature; structural modelling |
| Text blocks | `"""` | Clean replacement for SQL/XML string literals |
| Switch expressions | `case .*->` | Replaces statement-switch where a value is returned |
| Pattern-matching `instanceof` | `instanceof [Type] [var]` | Eliminates explicit casts after type checks |

#### Why it matters

Java 21 feature adoption is the primary **positive indicator** of the modernisation.
A codebase that still uses zero records, zero text blocks, and zero switch expressions after
Phase 1 has not materially modernised — it has only been compiled with a newer JDK.

#### How to interpret

- These are **cumulative totals across production source** (`api/src/main/java`, `web/src/main/java`).
- An increasing total is progress. A flat or decreasing total after a Phase 1 PR is a smell.
- The `var` counter is the easiest to move and the lowest risk. It should be non-trivially positive after Phase 1.
- Records and sealed classes have architectural implications; they move more slowly and deliberately.

### 3.4 Code Smells

#### What we capture

| Smell | Pattern | Risk |
|---|---|---|
| God files (> 400 lines) | `wc -l` on each `.java` file | High — SRP violation |
| God files (> 800 lines) | As above | Critical — multiple SRP violations |
| Broad `catch (Exception)` | Text match | High — hides real failure types |
| Catch `(Throwable)` | Text match | Critical — catches `Error`s and `OOM` |
| Silent catch blocks | Empty/comment-only catch body | High — failures silently swallowed |
| Boolean params in signatures | `boolean` in method parameters | Medium — method should be split |
| String `switch` dispatch | `case "..."` | Low-medium — enum preferred |

#### Why it matters

**God files** accumulate risk because:
- Every change to them has a wide blast radius.
- They are harder to test in isolation.
- They obscure ownership and responsibility.

**Broad/silent catches** are reliability killers in a healthcare system:
- `catch (Exception)` catches `RuntimeException` and checked exceptions alike — masking programming errors.
- `catch (Throwable)` will swallow `OutOfMemoryError` and prevent JVM crash signals from propagating.
- A silent catch block means an error occurred but no log entry, no metric, no alert. The system
  continues as if nothing happened.

**Boolean parameters** (`void save(Patient p, boolean includeVoided)`) violate the Command-Query Separation
principle and create callers that pass `true` or `false` with no self-documenting intent.

#### How to interpret

These are **regression gates**: the counts should never increase, and should trend downward.
A PR that adds a new god file (a new file > 400 lines, or an existing file that grows past 400)
should be flagged in code review.

### 3.5 Performance

#### What we capture (methodology only)

Build-time static analysis cannot measure runtime behaviour.
Performance baselines require a running instance and are captured manually at environment standup milestones.

| Metric | How to capture | When |
|---|---|---|
| Application startup time (cold) | `time docker compose up --wait` against the dev stack | Phase 2 milestone |
| Application startup time (warm) | Restart with image already pulled | Phase 2 milestone |
| REST API latency — concept search | `curl` with `time`, repeated 10×, take median | Phase 2 milestone |
| REST API latency — patient save | As above | Phase 2 milestone |
| Container image size | `docker image inspect openmrs-core --format '{{.Size}}'` | Phase 2 milestone |

#### Why it matters

The modernisation must not degrade end-user experience or system reliability.
Startup time is a leading indicator for deployment agility (critical in a containerised world).
Endpoint latency is a direct patient-impact signal.

#### Where to record results

When first captured, performance baselines should be committed to this document in a new section:
`## 4.x Performance Snapshot` with the date, environment, and Docker Compose service versions.

## 6. Regression Detection Strategy

### 6.1 What counts as a regression?

A regression is when a modernisation PR makes the codebase **measurably worse** on any tracked metric.
This is distinct from a bug — a regression is a structural or quality deterioration, not a functional failure.

| Regression Type | Example | Severity |
|---|---|---|
| New god file introduced | A new 600-line class is added | Medium |
| Existing god file grows | A 400-line file grows to 450 lines | Low |
| New broad catch added | `catch (Exception e) { log.error(e); }` in new code | Medium |
| Coverage drops | A refactored method has no test coverage | High |
| SpotBugs finding | A new bytecode-level bug pattern introduced | High 🔒 |
| New `@SuppressWarnings` | A new suppression added without explanation | Medium |
| `@Deprecated` annotation added | New code marked deprecated immediately | Low |

### 6.2 Automated regression gates (currently active)

These gates run in CI on every PR (`ci-modernisation.yml`):

| Gate | Implementation | Fails on |
|---|---|---|
| SpotBugs | `spotbugs:check` | Any new unsuppressed finding |
| Checkstyle | `checkstyle:check` | Any error-severity violation |
| Tests pass | `verify` | Any test failure |
| Coverage (future) | `jacoco:report` + threshold rules | Drop below configured minimum |

### 6.3 Snapshot-based regression detection

The `capture-baseline-metrics.sh` script supports snapshot comparison:

```bash
# Step 1: Save today's snapshot (do this before starting a phase)
bash tools/capture-baseline-metrics.sh \
    --snapshot docs/snapshots/pre-phase1-$(date +%Y%m%d).md

# Step 2: During the phase, compare current state against snapshot
bash tools/capture-baseline-metrics.sh \
    --diff docs/snapshots/pre-phase1-20250101.md
```

The diff output flags any metric that has moved in the wrong direction.
Exit code `2` means at least one regression was detected.

This can be integrated into CI as an informational check:

```yaml
- name: Check metrics regression vs baseline snapshot
  run: |
    bash tools/capture-baseline-metrics.sh --diff docs/snapshots/baseline.md || \
    echo "::warning::Metrics regression detected — see diff output above"
```

### 6.4 PR review checklist items

Every modernisation PR should confirm:

- [ ] God file count has not increased
- [ ] No new broad `catch (Exception)` added (or existing one was narrowed)
- [ ] No new `@SuppressWarnings` without a code comment justification
- [ ] No new `java.util.Date` usage (use `java.time` instead)
- [ ] Coverage has not decreased from the previous run
- [ ] If a Java 21 feature was added, the adoption counter went up

## 7. Interpreting the Numbers

### 7.1 What the counters can and cannot tell you

| Counter | Reliable for | Not reliable for |
|---|---|---|
| God file count | Tracking decomposition progress | Judging code quality of individual files |
| Broad catch count | Regression detection | Identifying which catches are actually problematic |
| `var` count | Adoption trend | Code readability of any specific site |
| Coverage % | Safety of refactoring | Correctness of the tests themselves |
| SpotBugs findings | Real bug patterns | Design quality |

### 7.2 Thresholds are minimums, not goals

The Phase 1 targets in [Section 5](#5-phase-1-targets) define the **floor** for Phase 1 completion.
They are not the end-state. A codebase where Phase 1 ends with exactly 100 `var` uses and 5 records
has done the minimum — the targets for Phase 3 are the more meaningful destination.

### 7.3 Metric gaming

Static analysis counts are easy to game.
Adding 100 `var` declarations to a single utility method inflates the counter without
reflecting genuine modernisation intent.
These numbers should be reviewed alongside PR diff statistics, not in isolation.

### 7.4 The JaCoCo gap

Until the JaCoCo baseline is captured (P0-08), coverage data is absent.
All other metrics in this document are available today from static analysis.
Coverage is the most important metric for refactoring safety, so P0-08 is a hard dependency
for beginning Phase 1.5 god-class decomposition.
